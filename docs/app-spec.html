🎯アプリの目的

SpeakNest は、日本人英語学習者向けのローカル動作英語学習アプリである。 インターネット接続を必要とせず、ローカル環境または LAN 内の自己ホスト型 AI サービスを利用して 以下の機能を提供する。

  • 🎤 音声会話練習 — マイクで話すと AI が英語で返答し、日本語訳も表示
  • 📝 フィードバック — 文法・発音に関する日本語フィードバックを自動生成
  • 📊 発音評価 — Word Error Rate (WER) ベースの発音スコアリング
  • 🌐 シナリオ学習 — カフェ注文、ホテルチェックイン等のシチュエーション別練習

📐データモデル

ChatRequest

{
  level:      str = "A2"           # CEFR レベル
  scenario:   str = "カフェで注文する"  # シナリオ
  transcript: str | None           # 発話テキスト（事前 STT 済み）
}

ConversationResponse

{
  transcript:    str  # STT 後のユーザー発話
  reply_en:      str  # AI 英語返答
  reply_ja:      str  # AI 日本語訳
  feedback_ja:   str  # 文法・発音フィードバック（日本語）
  next_practice: str  # 次の練習フレーズ提案
}

HealthResponse

{
  status:     "ok" | "degraded"
  ollama:     "ok" | "not_available"
  stt:        "ok" | "not_available"
  upload_dir: str
}

ConfigResponse

{
  configured: list[str]  # 設定済み環境変数名
  missing:    list[str]  # 未設定環境変数名
}

PronunciationRequest

{
  target_sentence: str        # 評価対象テキスト
  transcript:      str        # ユーザー発話テキスト
  sentence_mode:   bool = False  # 文末記号ゲート
}

PronunciationResponse

{
  wer:                  float      # [0.0, 1.0]
  missing_words:        list[str]  # 抜けた単語
  extra_words:          list[str]  # 余分な単語
  target_word_count:    int
  transcript_word_count:int
  accuracy:             float      # 1.0 - wer
  is_valid:             bool       # 両入力に単語あり
  llm_context:          dict       # LLM フィードバック生成用情報
}
⚠️
データベース (db/ ディレクトリ) は現在 未実装。会話履歴・学習進捗データの永続化は Phase 3 で SQLite 追加予定。現状は音声ファイルのみ data/recordings/ に保存される（一時ファイル・削除タイミングは要確認）。

🔄主な処理フロー

1. 音声会話フロー（メインフロー）

2. 発音評価フロー

1
入力受付

POST /pronunciation に target_sentence (お手本テキスト)、transcript (ユーザー発話)、sentence_mode を送信。

2
sentence_mode ゲート

sentence_mode=True かつ target が .!? で終わらない場合、評価をスキップ。

3
テキスト正規化

小文字化 → 句読点除去（内部アポストロフィ保持） → 空白整理。

4
WER 計算（Levenshtein）

単語レベルの編集距離を DP で計算。
WER = 編集距離 / max(target 単語数, transcript 単語数)

5
差分単語抽出

DP バックトレースで match / substitution / deletion (missing) / insertion (extra) を分類。

6
LLM コンテキスト生成

WER・accuracy・missing/extra words を辞書として LLM フィードバック生成用に返却。

3. ヘルスチェックフロー

GET /api/health
  │
  ├── ollama_available()  → GET {OLLAMA_BASE_URL}/api/tags  (timeout: 3s)
  │     OK? → ollama: "ok"
  │     NG? → ollama: "not_available"
  │
  └── stt_available()     → GET {STT_BASE_URL}/health      (timeout: 3s)
        OK? → stt: "ok"
        NG? → stt: "not_available"

status: "ok" ← 両方 OK のとき
status: "degraded" ← 一方でも NG のとき

起動順序（ヘルスチェック依存）

① whisper コンテナ起動
   └─ ヘルスチェック: GET /health  (30s間隔, timeout 5s, retries 3, start-period 300s)
② backend コンテナ起動  ← whisper が healthy になってから
③ frontend コンテナ起動 ← backend が started になってから
⚠️
Whisper の start-period: 300s (5分) は、初回の model ダウンロード時間を考慮したもの。2回目以降の起動ではもう少し早い。

🔐認証・認可

ℹ️
現在（Phase 1）は認証・認可の仕組みは一切実装されていない。
  • API キー・JWT トークン等は不使用
